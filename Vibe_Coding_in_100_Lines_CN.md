# 《Vibe Coding in 100+ Lines》：探索 Agent 设计另一种思路

我的第一份工作是游戏策划。除去撰写游戏策划设计方案外，我还有一个重要的工作内容，就是写游戏脚本。玩家玩到的游戏里的任务指引，任务结算，都来自我写的脚本。这些脚本都是我提前写好的。
游戏引擎是一个 Runtime，我写的脚本是在这个 Runtime 里运行，真正让玩家进行消费的内容。
带着这个思路，回头审视现在的 vibe coding，我有了一个想法。为了验证这个想法，我写了一个 mini 版本的 vibe coding 工具。写完最初版只有 100 多行代码。后来为了稳定性，效果等因素又扩充了一些代码。但让 ai 重新压缩后，依然可以压缩在 100 多行，含 prompt 的情况下实现一个 vibe coding 的核心功能。
我会基于这个 mini vibe coding 工具，展开说说我对未来一种Agent 设计的新的思路。

## 0. TL;DR

- **Runtime 优先于 Tools**  
  Agent 的能力边界，应该由它所处的执行环境（Runtime）决定，而不是由一长串工具接口来堆出来。

- **给 Agent 一个可组合的执行语义，而不是菜单**  
  与其让 Agent 在不同 tools 之间做选择，不如让它在一个统一、可组合的 Runtime 中直接生成并执行行为。

- **`bash` 是最小可行 Runtime 的现实范例**  
  文件、进程、管道、退出码，已经足以表达大多数工程动作，也恰好是 LLM 最熟悉的执行语言。

- **Agent 的职责不是“调用工具”，而是“把任务跑完”**  
  任务由人定义，约束由 Runtime 提供，Agent 只负责在反馈中推进执行，直到收敛。

- **真正的 Agent 闭环发生在 Runtime 中，而不在 prompt 里**  
  生成、执行、观察、修正，只有在可执行环境里，这条链路才成立。

## 1. 什么是 Runtime
我说的 Runtime，不是 JVM / Node / Python 这种“语言运行时”的狭义概念，也不是“模型本身就是 Runtime”。我想表达的是一种更产品化、更 agentic 的东西：

> **Runtime = 一个可被 Agent 驱动的执行环境。**  
> Agent 可以根据用户行为实时生成（或改写）代码，把它丢进这个环境里执行，并把执行结果再反馈回来，形成闭环。

它的关键在于：**Agent 交付的不再是一行行文字，一个个静态序列化的 json 对象，而是一个用户可以直接感知的结果。**

## 1.1 静态应用的本质：提前写死的交互

传统互联网产品（网站、App）的交互体验，本质是：

- 产品经理设计流程  
- 程序员提前写好页面 / 逻辑  
- 用户的每一次点击，都只能落在这套预先写死的分支上  

这套模式的优点是稳定、可控、可预测。缺点也同样明显：  
**它只能覆盖「你提前想到的用户行为」。**

用户的真实意图往往更丰富、更细微、更因人而异，但系统只能用有限的固定流程去接住它。

## 1.2 游戏里的 Runtime：脚本在运行时决定体验

我在做游戏策划的时候，写过大量任务脚本：任务触发、指引、结算、奖励发放……这些内容最终都要在游戏引擎里跑起来，玩家才能“体验到”。

从工程视角看，游戏引擎是 Runtime：

- 引擎提供能力边界（移动、战斗、对话、奖励、状态机……）  
- 脚本描述具体逻辑（什么时候触发什么、怎么推进、如何结算）  
- 脚本在运行时被解释 / 执行，最终表现为玩家看到的「内容与体验」  

但这里有个前提：**脚本是我提前写好的。**  
也就是说：即便游戏引擎是运行时，玩家体验仍然是被“预制”的。最多根据我的 if-else 走向不同，但还是预制的分支。

## 1.3 我想要的 Runtime：脚本不再预制，而是动态生成

我真正想讨论的是一个更激进的变化：

> **如果任务脚本不是策划提前写的，而是由 Agent 根据玩家的实时行为、偏好、状态，在运行时动态生成并执行呢？**

那游戏体验会发生质变：

- 同一个任务，不同玩家会得到不同的推进方式  
- 同一个关卡，系统会根据玩家能力动态调整目标、奖励、难度  
- NPC 的对话与行动不再是固定树，而是可控范围内的「生成式剧情」  
- 玩家每次登录，获得的任务都是「为他此刻量身定制」的  

这时，“脚本”从一个静态文件，变成一种**运行中的动态代码**。  
它是被生成、被执行、被验证、被迭代的。

## 1.4 把网站 / App 的静态代码变成动态化

同样的想法落到互联网产品，就变成：

- UI 不再完全由固定页面和固定按钮构成  
- 用户的行为、目标、上下文会驱动 Agent 动态生成交互逻辑  
- 这段逻辑不是“描述”，而是真的能跑：它会修改界面、调用工具、改变状态、给出下一步  

换句话说，过去我们做产品是：

> 写一套静态代码 → 覆盖尽可能多的用户路径

而现在我提出的是：

> 提供一个 Runtime → 让 Agent 在用户旅程中实时生成「下一段最合适的代码」，把体验动态跑出来

这会带来一种完全不同的产品交互：

- **交互更丰富**：因为不必把每条路径都提前 UI 化  
- **交互更准确**：因为它可以根据「这个用户此刻的意图」生成  
- **体验更个性化**：每个用户的流程可以不同，但仍可被约束、可验证、可回滚  


## 1.5 为什么这叫 Runtime，而不是“生成代码”

之所以强调 Runtime，是因为关键点是**「跑起来」**，而不是「写出来」。

- 只生成代码，用户看到的仍然是文本  
- 只有 Runtime 能执行，用户才会得到可感知的反馈  

例如：

- 页面状态改变了  
- 任务完成了  
- 数据被写入了  
- 结果可验证了  
- 出错能捕获并恢复  

Agent 的生成行为必须嵌入这条链路里：

> **生成 → 执行 → 观察 → 修正**

没有 Runtime，就没有闭环；  
没有闭环，Agent 很难稳定收敛到「可用的产品体验」。


## 1.6 为什么要做一个 100+ 行的 mini vibe coding 工具

我写的 mini vibe coding 工具，本质不是为了证明「100 行也能写个 AI coding」，而是为了验证一个更具体的假设：

> **Agent 能否在一个受控的 Runtime 里，基于反馈进行迭代式生成，并把「静态工程」动态化？**

也就是说，我真正关心的是：

- Runtime 提供哪些「可执行的原语」（执行、读写、回滚、测试、 diff）  
- Agent 如何根据用户反馈生成最小改动  
- 如何让每一步都**可见、可验证、可收敛**，而不是一次性重生成  


## 2. Runtime 下 Agent 的设计有什么优势？

在当前的 Agent 实践中，最常见、也最“顺手”的做法，是不断给 Agent **堆 tools**。

需要读文件？加一个 tool。  
需要写代码？再加一个 tool。  
需要跑测试？再加一个 tool。  
需要部署？继续加。

每一个新能力，都会带来一整套配套工程：

- 一个工具接口定义  
- 一份 schema  
- 一套权限与边界  
- 一组错误码与失败语义  

最终系统演化成的，往往不是一个“智能体”，而是一个**工具注册表**。

---

### 2.1 堆 tools 的问题，不是功能不够，而是语义碎裂

从表面看，tools 路径是合理的：

> Agent 不会做的事 → 给它一个工具 → 问题解决

但从 Agent 行为建模的角度看，这条路径有一个根本问题：

> **你得到的是一堆离散动作，而不是一个可组合的执行语义。**

每个 tool 都是一个“点状能力”：

- 读文件是一个点  
- 写文件是一个点  
- 跑测试是一个点  
- 调接口是一个点  

而 Agent 真正要完成的，是一段**连续的、可推演的行为过程**。  
当行为只能通过“选哪个 tool”来拼接时，复杂度会被不断放大。

---

### 2.2 Tools 路径带来的真实工程性退化

堆 tools 的危害，通常不是立刻显现，而是体现在系统逐渐复杂之后。

#### 1）上下文污染（Context Pollution）

为了让 Agent “可能用得上”某些能力，你不得不：

- 把越来越多的 tool 描述塞进 prompt  
- 包含参数说明、使用示例、错误处理规则  
- 甚至还要提示“什么时候不该用这个 tool”  

结果是：

- Token 被工具文档吞掉  
- 模型注意力被稀释  
- 真正和当前任务强相关的信息，反而变得不突出  

Agent 并不会因为 tools 更多而更聪明，它只会变得更犹豫。

---

#### 2）工具选择的不确定性，被转嫁给了编排层

在 tools 模式下，每次任务开始前，你都会隐式地面对这些问题：

- 这次要加载哪些 tools？  
- 不加载会不会缺能力？  
- 全加载会不会误用？  
- 不同任务之间，tool 集合是否应该变化？  

最终，你会发现：

> **你并没有减少复杂度，只是把复杂度从实现端，转移到了编排端。**

而这个编排端，往往是：

- 更难测试的  
- 更难推理的  
- 更容易变成“玄学调 prompt”  

---

### 2.3 Runtime 路径：先给一致的执行语义

Runtime 的设计路径，恰恰是反过来的。

它不是从“我要哪些能力”出发，而是从一个更基础的问题出发：

> **Agent 执行行为时，最小、最稳定、最可组合的原语是什么？**

在 Runtime 模型下：

- Agent 不需要关心“这个能力是不是一个 tool”  
- 它面对的是一套**一致的执行语义**  
- 所有复杂行为，都是由同一组原语组合而成的  

例如：

- 执行一段代码  
- 读取或修改状态  
- 观察执行结果  
- 基于反馈生成下一步  

工具不是不能加，但它们的角色发生了变化：

> **Tool 是 Runtime 的补充，而不是 Runtime 的替代。**

---

### 2.4 Runtime 天然带来的性能与成本优势

除了设计层面的优势，Runtime 路径在**性能和成本**上也更友好。

#### 1）更短、更干净的输入

在 Runtime 模式下：

- 不需要在 prompt 里反复描述 tools 的参数结构  
- 不需要让模型生成冗长、脆弱的 JSON，只是为了“匹配一个接口”  
- 不需要教模型“如何正确调用某个 API”  

Agent 只需要输出：

> **能被 Runtime 执行的代码或脚本**

这对 LLM 来说，是一个天然更友好的目标形式。

---

#### 2）代码 / 脚本比 API schema 更符合模型直觉

LLM 在训练中：

- 看过海量代码  
- 熟悉控制流、条件判断、状态变化  
- 天然擅长表达“先做什么，再做什么，失败怎么办”  

相比之下，人类设计的 API schema 往往是：

- 冗长的  
- 非语义化的  
- 为机器而非为模型设计的  

尤其是在**组合调用**场景下：

- tools 模式需要多轮调用、多次决策  
- Runtime 模式可以在一次输出中，表达完整的多步逻辑  
- 执行顺序、依赖关系、错误处理，都由代码自然表达  

---

### 2.5 从“选工具”到“写行为”的转变

这背后，其实是 Agent 设计范式的一个根本转变：

- Tools 路径：Agent 的主要决策是 **“下一步用哪个工具”**  
- Runtime 路径：Agent 的主要产出是 **“下一段可执行行为”**  

前者更像是在操作一个菜单系统，  
后者更像是在编排一个正在运行的系统。

也正因为如此，Runtime 模式更容易形成：

- 连续行为  
- 可回放的执行历史  
- 可 diff、可回滚、可验证的结果  

## 3. Runtime 型 Agent 为什么会是未来 Agent 的形态之一？

我认为 Runtime 型 Agent 并不是一个“通用解”，也不是要完全取代现在 Tools 的解法。而更像是一个**在复杂任务中，逐步进化出来的结果**。

---

### 3.1 现实约束一：Agent 必须运行在“已有系统”之中

Agent 从来不是运行在真空里的。

在真实世界中，它必须嵌入：

- 已存在的网站和 App  
- 已上线多年的业务系统  
- 结构复杂、风格各异的代码库  
- 不同团队、不同时期做出的工程决策  

一个经常被忽略的事实是：  
**世界不会为了 Agent 重写一遍。**

我们不会看到：

- 所有 Web / App 被统一成“Agent 友好”的交互形式  
- 所有系统 API 被重构成完美的工具接口  
- 每一个产品都为 Agent 量身定制一套能力抽象  

现实更可能是：

- 遗留系统长期存在  
- UI 和流程高度异构  
- 工程团队不可能为每一个 Agent 场景不断重构工具链  

在这种前提下，Agent 想要真正参与到产品中，必须有一个**缓冲层**。

Runtime 正是这样一个中间层：

- 人类系统只需要提供一个**可执行、可受控的环境**  
- Agent 不需要理解世界上所有系统的细节  
- 复杂性被“包裹”在 Runtime 内部  

从这个角度看，Runtime 并不是为了让 Agent 更自由，而是为了**让 Agent 能在现实世界中生存下来**。

---

### 3.2 现实约束二：Agent 的行为需要被“治理”，而不是被“限制”

在很多 Agent 设计中，安全与可控通常通过一种方式实现：

> **限制 Agent 能做的事情。**

于是我们会看到：

- 严格的 tools 白名单  
- 越来越复杂的参数校验  
- 越来越保守的能力开放  

这种方式在早期是有效的，但一旦 Agent 的能力开始扩展，就会遇到一个问题：

- 能力一多，限制就开始崩溃  
- 每加一个能力，就要重新思考边界  
- 系统复杂度随能力线性甚至指数增长  

Runtime 型 Agent 走的是另一条路。

它并不试图限制 Agent 的“行为形态”，而是通过 Runtime 本身来治理行为的**后果**：

- 统一的执行模型  
- 明确的权限与资源边界  
- 可观察的执行过程  
- 可回滚、可重放的结果  

这是一种非常工程化的思路：

> **不假设 Agent 永远做对，  
> 而是假设错误一定会发生，并且系统必须能承受。**

从这个意义上说，Runtime 更像是一种**治理结构**，而不是能力列表。

---

### 3.3 现实约束三：真实任务具有“过程性”，而不是一次性决策

在理想化的 Agent 设想中，任务往往被描述成：

> 给定一个目标 → Agent 规划 → 执行 → 完成

但在真实产品中，任务更像是：

- 执行一部分  
- 观察结果  
- 调整方向  
- 修正局部  
- 再继续执行  

这是一个**持续演化的过程**。

Tools 型 Agent 在这种场景下会暴露出明显问题：

- 每一步更像一次“重新开始的决策”  
- 上下文需要不断重新构建  
- 历史行为难以被当作一等公民来处理  

而 Runtime 型 Agent 天然更适合承载这种过程性：

- 行为被串联成一条连续的执行轨迹  
- 每一步都有明确的前后关系  
- 执行历史可以被 replay / diff / rollback  

一旦任务具有明显的“过程属性”，Runtime 几乎成为唯一合理的承载方式。

---

### 3.4 Runtime Agent 最先会出现在哪些场景？

如果 Runtime 型 Agent 是一条成立的路径，那么它**一定不会同时出现在所有地方**。

它最先出现的，一定是那些：

- 允许系统“自己变形”  
- 用户能理解动态行为  
- 执行反馈足够快的场景  

从当前技术条件来看，最可能的切入点包括：

#### 1）开发者工具与编辑器类产品

例如：

- IDE  
- 配置驱动系统  
- 内部开发工具  
- 工作流 / UI / 代码生成工具  

这些场景的共同特征是：

- 已经存在 Runtime  
- 行为可执行、可验证  
- 用户对“系统在变化”有心理预期  

#### 2）游戏、模拟与高交互系统

例如：

- 游戏任务系统  
- 沙盒模拟  
- 教学 / 训练系统  

这些系统本身就建立在 Runtime 思维之上，区别只在于：

- 过去：脚本由人写  
- 未来：脚本由 Agent 写  

#### 3）复杂流程型 SaaS（而非普通消费级 App）

例如：

- 数据分析平台  
- 营销自动化  
- 企业内部系统  
- 低代码 / 无代码平台  

这些产品的核心问题是：

- 用户需求高度定制  
- 静态 UI 无法覆盖真实流程  

Runtime Agent 在这里并不是“锦上添花”，而是可能成为**唯一可扩展的解法**。

可以明确的一点是：

> Runtime 型 Agent 不会最先出现在普通 ToC 应用中，  
> 而会先出现在那些**允许系统本身不断变化的地方**。

---

### 3.5 从“写代码”到“运行中的产品形态”

回到第一章提到的 Runtime 视角，我们可以看到一个更大的变化正在发生：

- 代码不再只是交付物  
- 产品不再是静态形态  
- 行为可以在运行时被生成、执行、修正  

当产品开始在 Runtime 中生成自己的行为时，Agent 就不再只是一个“外挂工具”，而是**产品结构的一部分**。

从这个角度看，Runtime 型 Agent 并不是在回答：

> “Agent 能帮我们多做什么？”

而是在回答：

> **“当产品本身开始运行时演化，Agent 应该以什么形态存在？”**

这也是我认为，Runtime 型 Agent 会成为未来 Agent 形态之一的根本原因。

## 4. 我是怎么设计 mini vibe coding 工具的？

基于前面关于 Runtime 型 Agent 的讨论，我选择用一个 **mini vibe coding 工具** 来验证这条路径是否真的可行。

这是一个非常适合验证的场景：

- 问题足够复杂（涉及理解需求、修改代码、验证结果）
- 反馈足够明确（代码是否能跑、diff 是否合理）
- 失败成本足够低（本地仓库、可随时回滚）

更重要的是，它让我可以**绕开 Runtime 构建本身的高成本**。

Bash 命令行天然就是一个成熟的执行环境：
- 能读写文件
- 能运行程序
- 能查看状态
- 能验证结果

而且几乎所有 LLM，都是 **Bash 命令的“老用户”**。

在这个设计里，我关心的不是“AI 能不能写更多代码”，而是：

> **Agent 能否在一个受控的 Runtime 中，  
> 基于真实反馈，稳定地推进任务，并最终收敛。**

---

### 4.1 单工具：只有 `bash(cmd)`

这是整个设计中**最关键、也是最激进的选择**：

> **只提供一个 runtime 入口：`bash(cmd)`**

没有文件读写工具、没有 diff 工具、没有测试工具、没有专用 API。

这样做带来的直接好处是：**能力表达彻底统一**。

- 读文件，是命令  
- 改文件，是命令  
- 跑测试，是命令  
- 看 diff，是命令  
- 组合复杂操作，也只是命令序列  

这在 Agent 设计上非常重要。

如果存在多个 tools，Agent 面临的其实是：
> “我该选哪个工具？”

而在单一 Runtime 入口下，Agent 面临的问题变成了：
> **“我该执行什么行为？”**

这是一个从“工具选择”到“行为生成”的根本转变。

---

### 4.2 严格协议：每一步必须是结构化 JSON

为了防止 Agent 演化成“只会说正确废话”的系统，runner 对输出做了严格约束。

每一步，模型**必须**输出结构化 JSON，并且只能处于以下阶段之一：

- `init`：补全或更新项目上下文（必要时生成或完善 `Agent.md`）
- `plan`：只允许列 checklist，`cmd` 必须为空
- `act`：执行 **1 条** bash 命令（读 / 改 / 跑）
- `verify`：执行 **1 条**验证命令
- `chat`：只解释、不改动
- `done`：请求人类确认完成

这个协议解决的是一个非常常见的失控点：

> **模型“说得非常对”，但没有做任何可验证的事情。**

通过阶段划分：

- 规划不能偷偷执行
- 执行不能混在解释里
- 验证不能省略
- 每一步都有明确语义

这让 Agent 的行为从“模糊输出”变成了**可审计的过程**。

---

### 4.3 一步一命令 + Git 回灌：让收敛发生在现实世界

在这个 runner 中，每执行一次命令，都会发生三件事：

1. 执行命令  
2. 自动收集 `git status`  
3. 自动收集 `git diff`  

并把这些信息：

- 注入回下一步上下文
- 写入 `logs/run_*.log`

这相当于**强制建立了真实反馈回路**。

它带来的变化非常明显：

- Agent 不能再假设“我已经改好了”
- 每一次修改都会被 diff 明确呈现
- 人类可以随时介入、评估、否决

从设计角度看，这一步的意义在于：

> **把 Agent 的收敛过程，  
> 从模型内部的“想象空间”，  
> 拉回到真实世界的状态变化中。**

---

### 4.4 强制 `TASK.md`：把短期状态外显化

在这个工具中，有一个看起来“很啰嗦”的要求：

> **每一步都必须完整更新 `TASK.md`，而且是重写，不是 append。**

这不是为了记录日志，而是为了**外显短期状态**。

`TASK.md` 承担的是一个非常工程化的角色：

- 当前目标是什么
- 已完成了哪些步骤
- 正在验证哪些假设
- 下一步准备做什么

你可以把它理解为一个**短期状态机**。

相比之下，纯依赖 chat 上下文的问题在于：

- 长度不可控
- 注意力会漂移
- 状态隐含在模型内部，无法被检查

而 `TASK.md` 把这些信息**拉回到仓库中**，成为可读、可 diff、可回滚的事实。

---

### 4.5 外部化状态：scratch + summary + 近期历史

即便有了 `TASK.md`，仅靠 chat 上下文依然是不稳定的。

因此这个 runner 明确做了一件事：  
**把状态拆分，并外部化。**

- `.agent_scratch.md`：  
  临时推理草稿，允许脏、允许反复修改

- `.session_summary.md`：  
  每隔 N 步生成一次压缩摘要（由 `SUMMARY_EVERY` 控制）

- `HISTORY_TURNS`：  
  只保留最近若干轮对话，其余用 summary 代替

但更重要的一点是：

> **Agent 不只是“被动接收上下文”，  
> 而是可以主动回忆和查证历史。**

在这个 runner 里，Agent 可以：

- 查阅 `logs/run_*.log`
- 对照历史 `git diff`
- 回看之前的 `TASK.md`

把历史行为当作**外部事实源**，而不是模糊记忆。

这一步的目的并不是让 Agent “更聪明”，而是让它**更稳定、更少重复犯错**。

---

### 4.6 最低成本护栏：denylist + timeout + output clip

最后一层，是一组非常“便宜”的安全护栏：

- **denylist**：  
  禁止高风险命令  
  `rm -rf`, `mkfs`, `dd`, `shutdown`, `reboot`, `curl`, `wget`, `nc`, `ssh`

- **timeout**：  
  单步命令超时（默认 `20s`）

- **output clip**：  
  输出截断（`OUTPUT_LIMIT`）

这些规则显然不完美，但它们的目标也很明确：

> **不是杜绝错误，而是避免灾难性错误。**

它们把“随手一把梭”的行为，  
压缩成“有限度、可观察、可中断的探索”。

---

### 小结：这不是一个聪明的 Agent，而是一个能活下来的 Agent

回头看这个 mini vibe coding 工具，会发现它并没有追求：

- 更复杂的规划
- 更强的推理
- 更多的能力

相反，它在不断做减法：

- 单一 Runtime
- 严格协议
- 强制反馈
- 外显状态
- 便宜护栏

这些设计共同指向一个目标：

> **让 Agent 的行为可以被执行、被观察、被修正，并最终收敛。**

这也是我认为 Runtime 型 Agent 值得被认真对待的原因。

它不是为了“展示 AI 有多聪明”，  
而是为了回答一个更现实的问题：

> **在一个真实、复杂、不可控的系统里，  
> Agent 要如何才能真正参与进来，并持续工作？**

基于以上设计，可以简单 100 多行的代码就能实现一个 vibe coding 绝大部分功能。开源代码在 https://github.com/tcjacker/Minimal-Agentic-Coding

## 结语：Runtime 不是答案，而是一种更合理的起点

回顾前面的讨论，可以看到一个逐渐清晰的轮廓：

- 我们并不是缺少更强的模型  
- 也不是缺少更多的工具  
- 真正缺少的，是一种**能让 Agent 在现实系统中长期工作的形态**

Runtime 型 Agent 并不是为了取代现有的工具体系，也不是为了让 Agent“无所不能”。  
它更像是一种工程上的让步：

> **承认系统一定复杂，  
> 承认行为一定会出错，  
> 承认变化一定发生，  
> 然后为这些现实提供一个可运行的结构。**

在这个结构里，代码不再只是静态交付物，而是运行中的媒介；  
Agent 不再只是调用工具的决策器，而是持续生成和修正行为的参与者；  
产品也不再是一个固定形态，而是一段在 Runtime 中不断演化的过程。

这并不意味着 Runtime 型 Agent 一定会成为“唯一正确的答案”。  
但在当前的技术条件下，它可能是**最不违背现实的一种选择**。

也正因为如此，我才会尝试用一个极简的方式，去验证这条路径是否真的可行——  
不是从一个宏大的系统开始，而是从一个 **100+ 行的 mini runtime + Agent 原型** 出发。

如果 Runtime 型 Agent 真的有未来，那它一定不是从“完美设计”开始的，  
而是从这种**最小、可运行、可被验证的尝试**开始。
