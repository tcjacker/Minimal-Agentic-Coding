# 《Vibe Coding in 100 Lines》：Agent 需要 runtime，而不是堆 tools

我先有一个想法：一个 Agent 的核心不应该是“挂了多少工具”，而应该是“它运行在什么 runtime 上”。然后我才写了一个极简 runner，用 `bash(cmd)` 这个最小 runtime 来验证它是否可行。

这里的“vibe coding”我指的是那种顺手、快速、边探索边实现的节奏。它的优势很明显，但它常见的副作用也很明显：上下文漂移、改动不可追溯、跑没跑过不清楚、出了问题不知道从哪一步开始变坏。

这篇文章不是展示提示词技巧，而是复盘一个更工程化的做法：用 runtime 承载能力，用约束把行为锁住，用审计让过程可回放。

## 0. TL;DR

- runtime > tools：优先选择一个可组合的执行环境，再决定要不要加工具。
- `bash` 是最小 runtime：文件/进程/管道/退出码足够表达大多数工程动作。
- 约束比“聪明”更重要：单步一命令、强制 `TASK.md`、强制 `git diff` 回灌、允许人类中断。
- Agent 要能“主动回忆”：不是只靠 chat 上下文，而是可检索日志与历史改动。

## 1. 为什么我说 runtime 比 tools 更重要

我不反对 tools，但我反对“用 tools 代替 runtime”。

堆 tools 的典型路径是：每要一个能力，就加一个工具接口，再加一套 schema、权限、边界、错误码，最后系统变成“工具注册表”。你得到的是一堆离散动作，而不是一个可组合的执行语义。

它的危害通常不是“功能不够”，而是工程性变差：

- 上下文污染：为了让 Agent“可能用得上”，你会把越来越多的工具说明塞进 prompt，token 被工具文档吞掉，模型注意力也被稀释。
- 工具选择不确定：每次任务开始前都要回答“这次要加载哪些 tools？不加载会不会缺能力？加载多了会不会误用？”最终把复杂度从实现端转移到了编排端。

runtime 的路径相反：先把 Agent 放进一个一致的执行语义里，让它用同一套原语组合出复杂行为。工具不是不能加，但工具应该是 runtime 的补充，而不是 runtime 的替代。

这也是我写这个 runner 的动机：不是为了证明 bash 多强，而是为了证明“只要 runtime 选对，tools 可以很少”。

## 2. 为什么选 bash 当最小 runtime

bash 的价值在于它不是“一个工具”，而是一套通用原语：

- 文件：读、写、搜索、替换、拼接
- 进程：运行、退出码、超时、并发（必要时）
- 管道：把输出接到下一步处理（可组合）
- 纯文本：让日志和审计天然可落盘

换句话说，`bash(cmd)` 不是“给 Agent 一个终端”，而是给 Agent 一个 runtime：它能生成命令、生成脚本、拼装工作流，完成复杂功能实现，而不必每次都从最底层代码开始写起。

我认为未来的 AI 服务也会往这个方向走：AI 不是只会写代码，而是会生成“操作命令/脚本/工作流”，把现有系统的能力编排起来。写代码当然仍然重要，但更多时候代码会变成“必要时才补齐的底层”，而不是每次都重写的默认选项。

## 3. 这个 runner 做了什么约束（以及为什么）

这里的实现很朴素，主要文件是：

- `vibe_runner.py`：完整版 runner（更可配置）
- `vibe_runner_compact.py`：压缩版 runner（现在是 191 行，“100 lines”更像一种态度：可读、可改、可 hack）

真正关键的是约束设计，而不是代码量。

### 3.1 单工具：只有 `bash(cmd)`

这是最重要的选择：不提供一堆花哨 tools，只提供一个 runtime 入口。

这样做带来一个直接好处：能力表达统一。读文件、改文件、跑测试、看 diff，都是命令；组合复杂操作，也是命令序列，而不是“跨工具的状态机”。

### 3.2 严格协议：每一步必须是结构化 JSON

runner 要求模型每一步输出严格 JSON，并且分阶段：

- `init`：补全/更新项目上下文（必要时生成/完善 `Agent.md`）
- `plan`：先列 checklist，`cmd` 必须为空
- `act`：执行 1 条 bash 命令（读/改/跑）
- `verify`：执行 1 条验证命令
- `chat`：只解释不改
- `done`：请求人类确认完成

这能压住一个常见失控点：模型“说得很对”，但没有做任何可验证的动作。

### 3.3 一步一命令 + Git 回灌：让收敛发生在现实世界

每执行一次命令，runner 都会把 `git status` + `git diff` 注入回上下文并写入日志（`logs/run_*.log`）。

这相当于把“真实反馈回路”强制化：

- 模型必须直面真实改动，而不是凭空想象“我已经修好了”
- 人类可以用 diff 评估每一步是否合理
- 回滚、定位、复盘都更简单

### 3.4 强制 `TASK.md`：把短期状态外显化

`TASK.md` 每一步都必须完整更新（重写而不是 append）。

这件事非常工程化：它把任务进度、假设、下一步从“模型脑内”拉到了仓库里。你可以把它当作短期 state machine，而不是靠 chat 上下文硬记。

### 3.5 外部化状态：scratch + summary + 近期历史

纯靠 chat 上下文很容易爆长度、漂移、忘细节，所以我们加了：

- `.agent_scratch.md`：临时推理草稿（允许脏）
- `.session_summary.md`：每隔 N 步写一条压缩摘要（`SUMMARY_EVERY`）
- `HISTORY_TURNS`：只保留最近若干轮对话，前面的用 summary 代替

但更关键的是：Agent 还应该能“主动回忆”。在这个 runner 里，它可以翻阅并检索之前的操作记录（例如 `logs/run_*.log`、`TASK.md`、以及每步注入的 `git diff`），把历史行为当作外部事实源来查证，避免遗忘与重复试错。

这不是为了“更聪明”，而是为了“更稳定”。

### 3.6 最低成本护栏：denylist + timeout + output clip

我们做了非常便宜但有用的护栏：

- denylist：挡住 `rm -rf`, `mkfs`, `dd`, `shutdown`, `reboot`, `curl`, `wget`, `nc`, `ssh`
- timeout：单步命令超时（默认 `20s`）
- clip：输出截断（`OUTPUT_LIMIT`）

这些规则不完美，但能把“随手一把梭”变成“有限度的可控探索”。

## 4. 案例：连连看里的两次 vibe coding（证明它不是纸上谈兵）

我不想用抽象论证结束，所以拿我们真实跑过的一个小任务当例子：`vibe_test_workspace/lianliankan.html`。

它很适合展示 runtime 思路，因为这类 UI/规则 bug 有两个特点：

- 很容易“看起来对”，但边界条件错
- 必须跑起来、看真实行为、做验证

两次 run 的目标分别是：

- 2026-02-20：匹配两块时先画一条细线动画，线连上后再消除方块  
  日志：`vibe_test_workspace/logs/run_20260220_113715.log`

- 2026-02-21：修 bug，相邻两块不需要画线（选中相邻也出现一道线是不对的）  
  日志：`logs/run_20260221_101836.log`，以及我们在 `.agent_scratch.md` 里对“相邻”的定义澄清

如果这是纯聊天式 vibe coding，你很容易得到一个“看起来像修了”的 patch；但在 runner 里，每一步都必须：

- 读现场文件
- 给出下一条命令
- 接受 `git diff` 的现实反馈
- 把进度写入 `TASK.md`

连 `isEmptyRow`/`isEmptyCol` 这种一不小心就写错端点排除条件的细节，也能在“可回放步骤”里被定位出来（见 `vibe_test_workspace/bug_report.txt`）。

这就是我想验证的事情：runtime + 约束 + 审计，能把 vibe coding 从“写作”变回“工程”。

## 5. 相比主流 vibe coding，它的优势到底是什么

我不认为它“更聪明”。它的优势是工程性：

1) 可审计  
一步一命令，输出写日志，改动落在 `git diff`。

2) 可复现  
短期状态在 `TASK.md`，项目事实在 `Agent.md`，而不是飘在聊天记录里。

3) 低耦合  
只依赖 OpenAI-compatible 的 `/chat/completions`，模型可替换；runtime 选 bash 也只是一个起点。

4) 可纠偏  
`Ctrl+C` 进入 control 模式，人类可以插手跑命令或直接 feedback，然后继续。

5) 抗遗忘  
summary/scratch 只是被动稳定，更重要的是可以主动翻阅历史记录来“回忆”。

## 6. 代价与局限（我不想回避）

- 单工具意味着复杂动作要拆成更多步，有时会比“全家桶 agent”慢
- denylist 会挡住一些真实需要的动作（例如拉依赖、抓网页），需要按场景调整
- CLI loop 不等于 IDE 内联体验，它更像一个可控的 pair loop

但这些代价是有意为之：我的目标不是“让 Agent 更像魔法”，而是让它更像工程师。

## 7. 快速上手（与仓库一致）

按 `README.md`：

```bash
python3 vibe_runner.py "创建一个 hello.py 并写入打印语句"
```

OpenAI / Qwen 的配置也在 `README.md` 里（`PROVIDER/BASE_URL/API_KEY/MODEL`）。

## 8. 结语：先选 runtime，再谈 tools

如果你也觉得 vibe coding 最大的问题不是“不会写”，而是“不可控”，我建议换个视角：先选一个 runtime，再加上约束和审计，让 Agent 的行为回到工程化轨道上。
